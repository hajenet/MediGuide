# MediGuide Development Plan

현재 앱은 `index.html` 하나로 구성된 클라이언트 전용 간호/의학 약어 퀴즈 앱입니다. 앞으로 기능을 확장하기 전에 한글 데이터 복구, 구조 분리, 학습 흐름 개선을 우선하는 것이 좋습니다.

## 1단계: 현 상태 안정화

- 깨진 한글 문구와 퀴즈 데이터를 원문 기준으로 복구한다.
- 사용하지 않는 `crypto-js` 의존성을 제거할지 결정한다.
- 비밀번호 기능의 목적을 명확히 정한다.
  - 단순 입장 제한인지
  - 실제 보호가 필요한지
- 브라우저에서 기본 동작을 확인한다.
  - 비밀번호 입력
  - 문제 풀이
  - 정답/오답 표시
  - 결과 화면
  - 다시 시작

## 2단계: 코드 구조 개선

- HTML, CSS, JavaScript를 분리한다.
  - `index.html`
  - `style.css`
  - `app.js`
  - `quiz-data.js` 또는 `quiz-data.json`
- 퀴즈 데이터와 화면 로직을 분리한다.
- `innerHTML` 사용을 최소화한다.
- 인라인 `onclick` 속성 대신 JavaScript에서 이벤트를 등록한다.
- 앱 상태 관리 구조를 정리한다.
  - 현재 문제
  - 점수
  - 오답 목록
  - 퀴즈 진행 여부

## 3단계: 학습 기능 강화

- 문제 랜덤 출제를 추가한다.
- 보기 순서 랜덤화를 추가한다.
- 틀린 문제만 다시 풀 수 있게 한다.
- 즐겨찾기 또는 중요 문제 표시 기능을 추가한다.
- 카테고리별 학습을 지원한다.
  - 순환기계
  - 호흡기계
  - 신장/비뇨기계
  - 응급/기본간호
  - 진료과 약어
- 해설 보기/숨기기 기능을 추가한다.
- 학습 진행률을 저장한다.

## 4단계: 사용자 경험 개선

- 모바일 화면을 최적화한다.
- 긴 문항과 해설의 줄바꿈을 개선한다.
- 진행률 바를 추가한다.
- 정답 선택 후 자동으로 다음 문제로 넘어가는 옵션을 검토한다.
- 결과 화면을 개선한다.
  - 점수
  - 정답률
  - 오답 수
  - 카테고리별 약점
- 다시 시작 시 `location.reload()` 대신 상태만 초기화한다.

## 5단계: 데이터 관리 방식 개선

- 문제 데이터를 JSON 파일로 분리한다.
- 문제마다 고유 ID를 추가한다.
- 문제 필드 형식을 표준화한다.

예시:

```js
{
  id: "cv-001",
  category: "순환기계",
  question: "순환기내과를 지칭하는 약어는?",
  options: ["CV", "GS", "NS", "OS"],
  answer: "CV",
  explanation: "CV는 Cardiovascular의 약어입니다."
}
```

- 향후 문제 추가가 쉬운 구조로 변경한다.
- 문제 검수용 체크리스트를 작성한다.

## 6단계: 보안 및 배포 방향 결정

- 비밀번호가 필요 없다면 제거한다.
- 단순 학습용이면 `localStorage` 기반 설정을 유지할 수 있다.
- 실제 접근 제한이 필요하면 서버 기반 인증을 검토한다.
- GitHub Pages 배포를 검토한다.
- README에 실행 방법, 기능, 데이터 구조를 문서화한다.

## 7단계: 고급 기능

- 오답 통계를 저장한다.
- 일일 학습 목표를 추가한다.
- 시험 모드와 연습 모드를 분리한다.
- 제한 시간 모드를 추가한다.
- CSV 또는 JSON 문제 가져오기를 지원한다.
- 관리자용 문제 편집 화면을 검토한다.
- PWA를 적용한다.
  - 홈 화면 설치
  - 오프라인 학습
- 다크 모드를 추가한다.

## 추천 우선순위

1. 한글 복구
2. 파일 분리
3. 퀴즈 데이터 구조화
4. 오답 복습 기능
5. 모바일 UX 개선
6. GitHub Pages 배포

현재 상태에서는 기능을 바로 늘리기보다 깨진 텍스트와 단일 파일 구조를 먼저 정리해야 이후 확장이 쉽다.
